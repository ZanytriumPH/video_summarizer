# ADR-002：Send API 并发模式升级（Vibe Coding 迭代方式）

**决策日期：** 2026-04-13  
**相关 ADR：** [ADR-001](5.大模型核心工作流架构.md#adr-001并行并发策略选型节点内-threadpoolexecutor-vs-图级-send-api)  
**执行方式：** Vibe Coding（AI 自主识别改进、灵活迭代）  
**目标状态：** 从节点内 ThreadPoolExecutor 升级为 LangGraph Send API 图级 Fan-Out

---

## 执行方式说明

采用 **Vibe Coding** 开发范式，而不是预先计划的四阶段框架：

- **自主识别**：每次迭代时，AI 持续检测当前代码的瓶颈、冗余和改进机会
- **灵活调整**：根据测试结果、性能指标和代码质量反馈，动态调整实现策略
- **有机演进**：不僵硬遵循预定计划，而是让架构自然地从 ThreadPoolExecutor 演变到 Send API
- **持续反思**：每个改动后重新评估系统状态，识别下一步最优的改进方向

**关键指标（持续监察）：**
- ✓ 所有现有测试通过率（反向指标：必须保持 100%）
- ✓ Checkpoint 体积趋势（目标：能否有机地减少 50%+）
- ✓ 代码复杂度度量（目标：State 合并逻辑可维护性）
- ✓ 可观测性深度（目标：单 chunk 级别的性能追踪可行性）

---

## 并发模式升级迭代指南 (Flexible Evolution)

### 主题 1：State 优化与存储解耦

**自主识别触发条件：**
- Checkpoint 文件大小监察：若单次会话 > 50MB，触发优化
- keyframes 字段占比分析：若 base64 部分 > State 90%，优先改进
- 内存使用趋势：若峰值内存持续上升，进行分离

**核心改进方向：**

1. **Keyframes 引用化**（灵活推进）
   ```python
   # 当前状态检查：keyframes 是否已大量占用状态？
   # 是 → 立即改进；否 → 暂不处理
   
   # 目标演变方向：
   keyframes: List[Dict]  # {"time": "00:15", "frame_file": "..."}
   keyframes_base_path: str  # 存储基路径，支持本地/S3 切换
   ```

2. **其他大字段检查**
   - `transcript` 是否需要分片存储？
   - `chunk_results` 是否支持增量 checkpoint？
   - 是否需要引入压缩机制？

3. **测试驱动反馈**
   - 每次改动后立即运行完整测试套件
   - 监察是否引入新的数据丢失风险
   - 评估改动的简洁性（越简单越好）

**无需预定时间表** - 根据实时监察指标动态调整

---

### 主题 2：Send API 图级拓扑演变

**自主识别触发条件：**
- 单 chunk 重试需求出现：若发现某个 chunk 失败导致整节点失败，触发改进
- 前端精细进度条需求：若用户/产品要求展示 per-chunk 进度，启动优化
- 数据流量分析：若多个 chunk 并行执行的冲突检测到，考虑隔离设计

**核心改进方向：**

1. **ChunkTaskState 设计**（按需迭代）
   ```python
   # 自主检查：当前 VideoSummaryState 是否过于臃肿？
   # 是 → 考虑状态隔离；否 → 保持现状
   
   # 若需改进，逐步演变为：
   class ChunkTaskState(TypedDict):
       chunk_id: str
       chunk_spec: Dict[str, Any]
       # ...其他独立字段
   ```

2. **Reducer 设计迭代**
   ```python
   # 首次尝试：检查是否需要深度 merge 逻辑
   # - latency_ms 是否会产生冲突？
   # - chunk_results 顺序是否需要保证？
   # 根据实际情况决定复杂度
   ```

3. **图拓扑逐步演变**
   - 首步：在现有 ThreadPoolExecutor 方案中添加 Send API 的条件编译开关
   - 次步：当所有前置条件满足时，尝试单个分支的 Send API 改造
   - 再步：根据单分支结果，决定是否全量迁移

**无需完全规划** - 根据触发条件和测试反馈有机推进

---

### 主题 3：测试覆盖与性能反馈

**自主识别触发条件：**
- 新增任何状态修改后：自动运行测试套件，评估是否引入回归
- Checkpoint 体积波动 > 5%：自动触发分析，找出原因
- 并行执行出现异常：立即添加对应的测试用例

**核心改进方向：**

1. **测试驱动的演变**
   ```python
   # 每次改动前后都执行：
   - 现有的 87+ generation/extraction 测试
   - 图级并行测试 (12 scenarios)
   - API 状态消息测试 (5 scenarios)
   
   # 若所有通过 → 改动安全，可继续
   # 若某个失败 → 立即回滚或修复
   ```

2. **性能基线建立**
   - 定期采样记录 checkpoint 大小
   - 监察内存占用趋势
   - 编译性能对比数据（仅在必要时）

3. **灵活的灰度策略**
   - 添加环境开关：`CONCURRENCY_MODE="threadpool"|"send_api"`
   - 先在开发环境验证
   - 若稳定，逐步开放给测试环境

**持续的质量反思** - 无需预定测试计划

**自主识别触发条件：**
- 每次重要改进后：自动反思当前架构决策，更新相关文档
- 代码复杂度突增：自动检查是否需要补充设计解释
- 新增特性或限制：立即记录到架构文件

**核心改进方向：**

1. **持续的架构文档更新**

   - README 中的架构章节随之演变

2. **技术梳理与沉淀**

## Vibe Coding 迭代周期

不采用固定的四周期划分，而是基于**自主识别**和**实时反馈**的有机迭代：

### 每次开发周期的反思清单

**代码提交前：**
- [ ] 所有现有测试通过？（反向指标，必须保持 100%）
- [ ] 是否发现代码冗余或可优化的地方？
- [ ] 这次改动是否为下一步改进铺平了道路？
- [ ] Checkpoint 体积或内存占用是否有异常变化？

**提交后定期检视（每周）：**
- [ ] 这一周的改进是否朝着 Send API 方向推进？
- [ ] 是否出现了新的改进机会（如新的重构点）？
- [ ] 测试覆盖率和代码复杂度是否维持在可接受范围？
- [ ] 是否需要调整优先级或改变策略？

**性能监察（持续）：**
```python
# 关键指标追踪
checkpoint_size_trend = [...]    # Checkpoint 体积是否在随时间减少？
memory_peak_trend = [...]         # 峰值内存是否稳定或下降？
test_execution_time = [...]       # 测试速度是否维持？
error_rate = 0.0%                 # 回归风险？
```

### 何时触发 Send API 迁移？

**早期信号（立即尝试）：**
1. 单个 chunk 失败导致整个节点失败 → 需要隔离设计
2. Checkpoint 文件 > 50MB for < 1 hour video → 需要存储优化
3. 需要展示单 chunk 级的进度条或错误信息 → 需要细粒度可观测性

**成熟条件（可以推进）：**
1. KeyFrames 存储已解耦为文件引用
2. 现有 ThreadPoolExecutor 方案已稳定运行 3+ 代视频
3. 有明确的性能基线数据支撑
4. 所有边界情况的测试都通过

**建议不推进（保持现状）：**
1. ThreadPoolExecutor 方案仍能满足所有需求（≤40min 视频）
2. 无新的业务需求压力
3. 团队人力/时间有限

## 自适应的风险管理

而非预先规划，而是边迭代边应对：

| 风险场景 | 自主识别方式 | 应对策略 |
|---------|-----------|--------|
| State 体积爆炸 | 监察 checkpoint 文件增长 | 若 > 50MB，立即分析并优化 |
| Reducer 逻辑 bug | 现有测试失败或数据校验和不符 | 立即回滚，补充单元测试 |
| 并行冲突 | 并发执行出现race condition | 添加锁/隔离，评估Send API必要性 |
| 性能回退 | 执行时间明显增加 | 性能分析，找出瓶颈，若无法优化则放弃该改动 |
| 测试覆盖不足 | 运行新测试时发现边界case | 立即补充对应的测试用例 |

**核心原则：每次发现问题时，实时调整策略；无需提前预案。**

### 当前已可讲述的亮点：

> **架构决策与权衡分析：**
> - 理解并权衡了 ThreadPoolExecutor（简单、低内存）vs Send API（细粒度控制）的关键差异
> - 基于当前视频处理规模（≤40 min），做出了合理的技术选型
> - 记录决策过程和 trigger conditions，展示工程化思维

> **并行执行设计：**
> - 实现了两层并行：图级 fan-out + 节点内 ThreadPoolExecutor
> - 设计并验证了 `_merge_chunk_results()` reducer 函数，处理并行分支的状态合并
> - 确保了顺序一致性和零数据丢失（通过 12+ 个单元测试验证）
