##### 5.3.1 当前实现状态（2026-04-14）
- `Audio_Agent` 与 `Vision_Agent` 已支持 `Send API` 图级 fan-out。
- 分片实时进度展示已落地，前端可实时看到音频与视觉分支进度。
- `chunk_synthesizer` 当前仍为节点内 `ThreadPoolExecutor` 并发实现，尚未迁移到图级 `Send API`。

##### 5.3.2 后续迁移目标：Synthesizer 全量对齐 Send API
本阶段的目标不是重写整个 5.3 架构，而是将 `chunk_synthesizer` 从“节点内线程池并发”迁移为“图级 Send API 分片融合”，形成完整一致的 Map-Reduce 闭环：

1. `chunk_planner_node`
2. `map_dispatch_node`
3. `chunk_audio_worker_node` / `chunk_vision_worker_node`
4. `synthesis_barrier_node`（方案 A 的中间汇聚门）
5. `route_synthesis_send_tasks()`
6. `chunk_synthesizer_worker_node`
7. `text_analyzer_node` / `vision_analyzer_node`
8. `fusion_drafter_node`

##### 5.3.3 设计决策（已确认）
**决策 1：图拓扑采用方案 A（Barrier / Gate Node）**

原因：
- `synthesis` 的分发时机晚于 `map_dispatch`，它不是初始 fan-out，而是第二阶段 fan-out。
- 在 `audio_worker` 与 `vision_worker` 之后显式引入一个中间汇聚门，更容易表达“等待上下游结果就绪后再进行二次分发”的语义。
- 有利于后续继续扩展 chunk 级重试、chunk 级可观测性和 chunk 级 checkpoint 恢复。

**决策 2：触发时机保底等待 audio_worker 和 vision_worker 整理完所有 chunk 后，再交付到 synthesizer 分发路由**

原因：
- 避免 `synthesis` 路由过早触发，导致部分 chunk 只拿到单模态结果。
- 降低重复路由和幂等控制复杂度。
- 优先保证工程稳定性与可解释性，而不是追求更激进的流水线重叠执行。

这意味着本轮迁移采取**保守一致性优先**策略：先让 `audio/vision` 两条 worker 分支把所有 chunk 整理完整，再统一进入 `synthesis` 分发阶段生成 `chunk_summary`。

##### 5.3.4 推荐图拓扑（方案 A）
```text
START
    -> chunk_planner_node
    -> map_dispatch_node
    -> chunk_audio_worker_node (Send fan-out)
    -> chunk_vision_worker_node (Send fan-out)
    -> synthesis_barrier_node
    -> route_synthesis_send_tasks()
    -> chunk_synthesizer_worker_node (Send fan-out)
    -> text_analyzer_node / vision_analyzer_node
    -> fusion_drafter_node
    -> graders -> END
```

其中：
- `synthesis_barrier_node` 不负责业务计算，只承担“所有分片结果已汇聚完成，允许进入二次路由”的门控职责。
- `route_synthesis_send_tasks()` 只对具备 `audio_insights + vision_insights` 且尚未生成 `chunk_summary` 的 chunk 下发 `Send` 任务。

##### 5.3.5 具体改造范围（Vibe Coding 落地清单）

- 新增或预留 `current_synthesis_chunk`、`current_synthesis_base_item` 作为 `chunk_synthesizer_worker_node` 的上下文字段。

**2. 分发层：** `core/workflow/video_summary/nodes/`
- 新增 `synthesis_barrier_node`。
- 新增 `route_synthesis_send_tasks()`。
- `route_synthesis_send_tasks()` 只在 `send_api` 模式启用。

**3. 融合层：** `core/workflow/video_summary/nodes/chunk_synthesizer.py`
- 保留现有 `chunk_synthesizer_node`，继续服务于 `threadpool` 路径。
- 新增 `chunk_synthesizer_worker_node`，每次只处理单个 chunk。
- 复用 `_process_single_chunk_synthesis()` 作为单 chunk 纯函数，避免逻辑分叉。

**4. 图拓扑：** `core/workflow/video_summary/graph.py`
- `threadpool` 路径保持不变。
- `send_api` 路径改为：audio/vision worker -> barrier -> synthesis send routing -> synthesis worker。

**5. 工作流播报与进度：** `core/workflow/api.py`

- 总体分母从 `2 * total_chunks` 升级为 `3 * total_chunks`。
- 前端新增“融合分片进度条”。

**6. 前端展示：** `app.py`
- 保留现有音频、视觉、总体进度条。
- 新增 `synthesis` 进度条，形成四段可视化：音频 / 视觉 / 融合 / 总体。

##### 5.3.6 触发条件与幂等规则
为避免重复派发 `synthesis` 任务，路由函数应满足以下条件：

1. `chunk_id` 非空。
2. `audio_insights` 非空。
3. `vision_insights` 非空。
4. `chunk_summary` 为空。
5. 仅在 `send_api` 模式生效。

本轮实现额外增加一个保守门槛：

6. 仅当 `audio_worker` 与 `vision_worker` 已完成全部 chunk 的整理后，`synthesis_barrier_node` 才允许进入 `route_synthesis_send_tasks()`。

该规则的目标不是追求最激进的并发，而是优先保证：
- `chunk_summary` 输入完整；
- 路由语义稳定；
- 前端进度展示可解释；
- 回归测试更容易写成确定性断言。

##### 5.3.7 分阶段实施计划
**阶段 1：最小后端闭环**
- 增加 `synthesis_barrier_node`。
- 增加 `route_synthesis_send_tasks()`。
- 增加 `chunk_synthesizer_worker_node`。
- 在 `graph.py` 中串起 send_api 路径。

**阶段 2：前端与可观测性闭环**
- 在 `api.py` 增加 `synthesis_done_ids`。
- 在 `app.py` 增加第四条融合进度条。
- 更新状态播报文案，使 send_api 路径形成完整的分片级可视化。

**阶段 3：测试闭环**
- 单元测试：worker、router、barrier。
- 集成测试：图级 fan-out / fan-in / synthesis 路由时机。
- 状态消息测试：融合进度与总体进度分母更新。

##### 5.3.8 验收标准
迁移完成后，应满足以下标准：

1. `threadpool` 路径零回归。
2. `send_api` 路径下，audio / vision / synthesis 三阶段均为 chunk 级任务。
3. 前端可实时展示音频、视觉、融合、总体四段进度。
4. `chunk_synthesizer_worker_node` 仅处理单 chunk，不再在节点内部使用线程池。
5. 所有 `chunk_summary` 均基于完整的双模态输入生成。

##### 5.3.9 风险与回滚策略
- 若 `synthesis` 路由触发次数异常，优先检查 barrier 条件与 chunk 去重逻辑。
- 若性能未提升，不视为本阶段失败；本阶段的首要目标是**架构一致性、chunk 级可观测性与后续恢复能力准备**。
- 如出现不稳定，可立即回退到现有 `threadpool` 默认路径，因为旧 `chunk_synthesizer_node` 将继续保留。
